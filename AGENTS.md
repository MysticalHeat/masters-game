## Repository Guide for Agentic Coders

### Project overview
- Unity project targeting **Unity 6000.3.10f1**.
- Main gameplay code lives under `Assets/Scripts/`.
- Editor-only tooling lives under `Assets/Editor/`.
- Runtime assembly compiles as `Assembly-CSharp`.
- Editor assembly compiles as `Assembly-CSharp-Editor`.
- C# language version is **9.0** and target framework is **netstandard2.1**.
- Important packages in use: Input System, URP, Sentis / Inference Engine, Unity Test Framework.

### Repository-specific instruction files
- No repo-local Cursor rules were found:
  - no `.cursorrules`
  - no `.cursor/rules/`
- No repo-local Copilot instructions were found:
  - no `.github/copilot-instructions.md`
- This `AGENTS.md` is the primary agent instruction file for the repository root.

### Files and directories you should treat as generated
- Do **not** hand-edit generated Unity project files unless the user explicitly asks:
  - `*.csproj`
  - solution files generated by Unity
- Do **not** edit Unity cache/build output folders:
  - `Library/`
  - `Temp/`
  - `Logs/`
  - `obj/`
- Avoid editing `Packages/packages-lock.json` unless dependency changes are intentional.

### High-signal project layout
- `Assets/Scripts/AI/` - local language model abstractions and Sentis integration.
- `Assets/Scripts/Gameplay/` - player, NPC, and chat loop gameplay logic.
- `Assets/Scripts/UI/` - Unity UI controllers and prompt views.
- `Assets/Editor/` - scene builder/editor automation.
- `Assets/Scenes/` - scenes, including the MVP scene created by tooling.
- `Packages/manifest.json` - Unity package manifest.
- `ProjectSettings/` - Unity editor and player settings.

### Preferred build / verification commands

#### 1) Main managed-code build
- Use this as the default compile check:

```bash
dotnet build "Assembly-CSharp-Editor.csproj"
```

- Why: this builds the editor assembly and also compiles the runtime assembly graph.
- This succeeded during repository analysis.

#### 2) Runtime-only build
- Use when you only changed runtime code and want a narrower compile pass:

```bash
dotnet build "Assembly-CSharp.csproj"
```

#### 3) Important build note
- Do **not** run the runtime and editor `dotnet build` commands in parallel.
- They share Unity-generated `Temp/obj` state and can race/fail.

#### 4) Lint / static analysis
- There is no dedicated repo lint script or formatter config.
- The practical lint/static-analysis command is the editor build:

```bash
dotnet build "Assembly-CSharp-Editor.csproj"
```

- Unity analyzers run as part of this build.

### Unity test commands
- The repository includes `com.unity.test-framework`, but **no committed tests were found** yet.
- There are also no test asmdefs in the repo right now.
- When tests are added, use the Unity Editor CLI, not `dotnet test`.

#### Set the Unity editor path once
```bash
export UNITY_EDITOR="/home/nomli/Unity/Hub/Editor/6000.3.10f1/Editor/Unity"
```

#### Run all EditMode tests
```bash
"$UNITY_EDITOR" -batchmode -nographics -projectPath "/home/nomli/masters game" -runTests -testPlatform EditMode -testResults "Logs/editmode-results.xml" -logFile "-"
```

#### Run all PlayMode tests
```bash
"$UNITY_EDITOR" -batchmode -projectPath "/home/nomli/masters game" -runTests -testPlatform PlayMode -testResults "Logs/playmode-results.xml" -logFile "-"
```

#### Run a single test
```bash
"$UNITY_EDITOR" -batchmode -projectPath "/home/nomli/masters game" -runTests -testPlatform EditMode -testFilter "Namespace.ClassName.TestName" -testResults "Logs/single-test-results.xml" -logFile "-"
```

#### Run a single test class
```bash
"$UNITY_EDITOR" -batchmode -projectPath "/home/nomli/masters game" -runTests -testPlatform PlayMode -testFilter "Namespace.ClassName" -testResults "Logs/single-class-results.xml" -logFile "-"
```

#### Test command notes
- `-testFilter` is the key flag for running one test or one class.
- `-nographics` is useful on headless machines; omit it if PlayMode tests need graphics.
- `-logFile "-"` streams logs to stdout.
- Prefer `Logs/*.xml` for results output.
- Do not assume tests exist; first confirm there are actual test files.

### Coding style observed in this repository

#### General formatting
- Use **4 spaces** for indentation.
- Use **Allman braces**:
  - opening brace on its own line
  - same for classes, methods, properties, control flow
- Use block-scoped namespaces, not file-scoped namespaces.
- Keep one logical statement per line.
- Prefer short blank-line-separated blocks for readability.

#### Imports / using directives
- Keep `using` directives at the top of the file.
- Order observed in repo:
  1. `System*`
  2. project namespaces like `MastersGame.*`
  3. Unity namespaces
- Remove unused usings.
- Do not use fully qualified names inline when a normal `using` is clearer, except for one-off attribute usage like `[System.Serializable]`.

#### Naming conventions
- Namespaces: `MastersGame.<Area>`.
- Types, methods, properties, enums: **PascalCase**.
- Private serialized fields: **camelCase** with `[SerializeField]`.
- Private non-serialized fields: **camelCase**.
- Local variables and parameters: **camelCase**.
- Constants and `static readonly` fields are currently written in **PascalCase** in this repo; follow the existing convention.
- Prefer descriptive names over abbreviations unless the domain term is standard (`npc`, `ui`, `kv`).

#### Types and API design
- Prefer explicit types for public APIs.
- Use `var` when the right-hand side makes the type obvious.
- Favor small DTO-like classes with get-only properties for request/response-style data (`ChatRequest`, `ChatMessage`).
- Prefer interfaces for swappable behavior (`ILocalLanguageModel`).
- Keep Unity-facing MonoBehaviours focused on orchestration and scene wiring.
- Put editor-only code under `Assets/Editor/`.

#### Unity-specific conventions
- Use `[SerializeField] private` for inspector-assigned references and tuning values.
- Use `Configure(...)` methods when objects are wired programmatically by editor tooling.
- Use `RequireComponent` where component dependencies are mandatory.
- Keep runtime gameplay logic out of `Assets/Editor/`.
- Prefer `Reset`, `Awake`, `OnEnable`, `OnDisable`, `OnDestroy` appropriately for lifecycle work.
- Unsubscribe event handlers in `OnDisable` or `OnDestroy`.
- Dispose unmanaged / disposable Unity resources where needed (`Worker`, tensors, cancellation sources).

#### Control flow and readability
- Prefer early returns to reduce nesting.
- Guard null or invalid state first.
- Keep methods focused and reasonably short when possible.
- Extract helper methods for repeated logic such as prompt cleanup, cache updates, UI builders, or logging helpers.
- Prefer clear state transitions over clever one-liners.

#### Error handling
- Validate required references before use.
- For user-triggered flows, fail gracefully with a usable status/message when possible.
- For unexpected exceptions, log with `Debug.LogException(exception)`.
- Catch `OperationCanceledException` separately for cancellable async flows.
- Preserve cancellation semantics by calling `ThrowIfCancellationRequested()`.
- Do not swallow exceptions silently.

#### Async and cancellation
- Async patterns in the repo use `Task` and `CancellationToken`.
- Pass cancellation tokens through async boundaries.
- Cancel outstanding work when UI/session state closes.
- Dispose `CancellationTokenSource` instances after use.
- Use `await Task.Yield()` only when it helps keep Unity responsive during long operations.

#### Collections and immutability
- Prefer `readonly` fields for long-lived collections.
- Favor `IReadOnlyList<T>` in public method signatures when callers should not mutate input.
- Trim history/state deliberately instead of letting it grow unbounded.

#### Logging
- Use `Debug.Log` for intentional debug traces.
- Gate noisy logs behind serialized debug toggles like `enableDebugLogging`.
- Prefix logs with the component name in brackets.
- Do not leave excessive spammy logging enabled by default unless it is already part of the workflow.

### What to do when adding tests
- Prefer creating dedicated EditMode or PlayMode test files instead of mixing tests into runtime scripts.
- Prefer a dedicated test assembly definition if you introduce a meaningful test suite.
- Name tests so they are easy to target with `-testFilter`.
- If only one behavior changed, add the narrowest useful test first.

### Practical agent guidance for this repo
- Start by checking whether a change belongs in `AI`, `Gameplay`, `UI`, or `Editor`.
- If a change touches scene construction or editor automation, inspect `Assets/Editor/NpcChatMvpSceneBuilder.cs`.
- If a change touches conversation flow, inspect `NpcChatGameManager`, `NpcChatTarget`, and the `ILocalLanguageModel` implementations together.
- If a change touches UI behavior, inspect both `ChatWindowController` and scene-builder code that instantiates the UI.
- Keep changes aligned with the current architecture; avoid introducing extra frameworks or package churn for small tasks.
